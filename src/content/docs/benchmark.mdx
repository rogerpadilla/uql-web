Our [open-source benchmark](https://github.com/rogerpadilla/ts-orm-benchmark) measures **pure SQL generation speed**, with no database I/O, to isolate the overhead each tool adds to every query. This page always reflects the latest run; the [original announcement post](/blog/in-search-of-the-fastest-typescript-orm) tells the full story.

> Node.js v24, Apple Silicon M-series, July 2026. Latest stable of every entry: TypeORM 1.0.0, MikroORM 7.1.5, Sequelize 6.37.8, Drizzle 0.45.2, Knex 3.3.0, Kysely 0.29.3, UQL 0.15.4. All values in **ops/sec** (higher is better), averaged over 3 runs.

## Results

| Query Type | [UQL](https://uql-orm.dev) | [Sequelize](https://sequelize.org) | [TypeORM](https://typeorm.io) | [MikroORM](https://mikro-orm.io) | [Drizzle](https://orm.drizzle.team) | [Knex](https://knexjs.org) | [Kysely](https://kysely.dev) |
| :--- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| INSERT (10 rows) | 473K | 208K | 42K | 114K | 13K | **487K** | 197K |
| UPDATE (SET+WHERE) | **1,761K** | 241K | 294K | 229K | 80K | 719K | 827K |
| UPSERT (ON CONFLICT) | **612K** | 340K | 269K | 272K | 37K | 445K | 342K |
| DELETE (WHERE) | **3,282K** | 1,370K | 527K | 267K | 212K | 1,102K | 1,276K |
| SELECT (1 field) | **3,349K** | 3,108K | 613K | 585K | 233K | 1,151K | 1,574K |
| SELECT (WHERE+SORT+LIMIT) | **1,161K** | 411K | 280K | 76K | 61K | 619K | 429K |
| SELECT (complex $or) | **619K** | 157K | 164K | 28K | 35K | 250K | 219K |
| AGGREGATE (GROUP+HAVING) | **1,527K** | 415K | 292K | 73K | 72K | 329K | 217K |

UQL is the fastest entry in 7 or 8 of the 8 categories depending on the run, on average **~2.1x** faster than the next fastest tool. Batch INSERT is a genuine statistical tie with the standalone query builder **Knex**: across repeated runs the two land within ~3% of each other and either can come out ahead (this run: Knex 487K vs. UQL 473K), unlike the other 7 categories, where UQL's lead is large and holds steady across every run. Explore the [interactive charts](https://rogerpadilla.github.io/ts-orm-benchmark/chart.html).

| Operation group | Avg advantage (vs. next fastest) |
| :--- | ---: |
| Reads (`SELECT*`, sort/limit, `AGGREGATE`) | **~2.3x** |
| Writes (`UPDATE/UPSERT/DELETE`) | **~2.0x** |

:::tip
**Query Builder vs. ORM**: The benchmark includes Knex and Kysely (pure query builders) to show that UQL is at least as fast as tools without the overhead of entities and relations, and outright faster in every category except the INSERT toss-up with Knex.
:::

## Why UQL is fast

Two design choices drive the numbers:

- Schema metadata (tables, columns, relations) is pre-computed once at startup, so no lookups happen at query time.
- SQL is written directly into a string buffer, avoiding intermediate builder objects and garbage-collection pressure.

## Methodology

- **What's measured**: pure SQL string generation, with no database, no I/O, and no connection pool.
- **Fairness**: every entry defines the same `User` entity, generates logically equivalent queries, compiles the PostgreSQL dialect, and uses its idiomatic API (QueryBuilder for TypeORM/MikroORM, `QueryGenerator` for Sequelize).
- **Why no Prisma?** Prisma's query compiler is Rust compiled to WebAssembly, not pure JS/TS, and it exposes no public SQL-compilation API, so it cannot be measured this way.
- **Reproduce it**: no database needed; it runs in seconds.

```bash
git clone https://github.com/rogerpadilla/ts-orm-benchmark.git
cd ts-orm-benchmark
bun install
npm run bench
```

The full methodology, fairness guarantees, and per-run data live in the [benchmark repository](https://github.com/rogerpadilla/ts-orm-benchmark).
